## 会话复盘

### 发生了什么
- 本轮围绕 94 服务器 aiclient2api 的用量查询、Codex 邮箱显示、potluck 本地统计回填、容器部署一致性做了多次代码与数据修复。
- 代码侧最终停在本地 `edcd2ca`，94 容器镜像也为 `edcd2ca`；最后一次用户要求尽量只修数据，因此只修改了 94 的 `usage-cache.json`。
- 期间暴露了 PowerShell/SSH/Bash 引号、缓存 TTL、容器代码一致性验证、以及“官方额度数据”和“potluck 本地统计数据”边界问题。

### 高信号问题
- PowerShell 会提前解析远端 Bash 片段：例如 `$(date ...)`、管道、正则和嵌套引号曾被本地 PowerShell 吃掉，导致远端 `docker rename/run` 命令异常。
- 只修生产数据时必须先确认数据来源：94 的 `usage-cache` 没有蜂智pro1成功额度快照，不能伪造官方额度；只能从 `model-usage-stats.json` 回填本地 token 统计，并标记 `staleUsage/lastRefreshError`。
- 修改 `usage-cache.json` 后还要考虑 TTL：缓存 timestamp 过旧会触发真实刷新并覆盖数据修复；只改数据时也可能需要同步更新 timestamp。
- 前端展示逻辑影响数据修法：`instance.error` 优先显示错误，会遮挡 `usage` 详情；要展示本地统计且保留异常，应把错误放到 `lastRefreshError`，并设置 `staleUsage=true`。
- 容器一致性验证不能只看镜像名或 SHA256：Windows CRLF 与容器 LF 会导致原始 hash 不一致；全量比对脚本也可能因为 `git archive` 参数或 CRLF 文件列表失败后误判。
- 最近一周同邮箱多条 provider 记录不是单纯数据错误：同一账号跨 `openai-codex-oauth` / `openaiResponses-custom`，以及重新授权/迁移 uuid，会导致 provider 维度重复。

### 下次默认做法
- 远端复杂命令默认使用临时脚本上传执行，或严格单引号包裹整段 SSH 命令；避免在 PowerShell 字符串中直接写 `$(...)`、复杂管道和 here-doc。
- 生产数据修复必须先只读扫描、输出脱敏摘要、备份目标文件，再做单实例 patch；不得凭空构造官方额度。
- 用 potluck 修 usage-cache 时，明确标记来源为本地统计，不把它当 OpenAI 官方额度；保留原始错误到 `lastRefreshError`。
- 修缓存类文件后检查 TTL/timestamp，否则 UI 可能立即绕过缓存触发真实刷新。
- 验证 94 与本地一致时优先比较运行时关键文件，或从 `git archive HEAD` 构建镜像并记录 commit tag；全量 tracked 文件比较要处理 Docker build context、CRLF 和测试文件未进入镜像的差异。
